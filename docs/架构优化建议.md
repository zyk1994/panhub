# PanHub 架构优化建议

> 架构师视角 | 基于当前代码库梳理

---

## 一、高优先级（建议尽快处理）

### 1. useSearch 冗余代码：performFastSearch 未被使用

**现状**：`performFastSearch`（约 334–410 行）已实现但与主流程脱节，主搜索只走 `performParallelSearch`。两套逻辑并存，增加维护成本和理解难度。

**建议**：二选一
- **方案 A**：删除 `performFastSearch`，保持单一入口 `performParallelSearch`
- **方案 B**：改造主流程为「快搜 → 深搜」分阶段，先 `performFastSearch` 再 `performDeepSearch`，实现更清晰的 Fast/Deep 分层

---

### 2. continueSearch 与 performParallelSearch 的职责不一致

**现状**：`continueSearch` 调用 `performDeepSearch`，后者只处理「剩余批次」（`slice(conc)` 之后的插件/TG）。而 `performParallelSearch` 会执行全部任务，暂停时已完成一部分。两者对「剩余」的界定不一致。

**建议**：统一策略
- 若采用「全部并行」：`continueSearch` 应继续执行 `performParallelSearch` 中尚未完成的任务，或记录已执行任务，从断点续跑
- 若采用「Fast/Deep 分阶段」：主流程改为 Fast + Deep，`performParallelSearch` 退化为 Deep 阶段实现，职责更清晰

---

### 3. extractMergedFromResponse 位置与复用

**现状**：`extractMergedFromResponse` 定义在 composable 内部，逻辑较重（75+ 行），与搜索逻辑耦合。

**建议**：
- 抽离到 `server/core/utils/` 或 `shared/utils/`，与 `SearchResponse` 类型一起维护
- 服务端若也需兼容多种响应格式，可共用该工具
- 单独为 `extractMergedFromResponse` 编写单元测试

---

### 4. useSettings 的 defaultTgChannels 初始化时机

**现状**：`settings` 初始化时使用 `defaultTgChannels.value`，而 `defaultTgChannels` 是 `computed`，依赖 `config`。SSR 或首次挂载时 config 可能尚未就绪。

**建议**：
- 在 `loadSettings` 或 `onMounted` 中再根据 `defaultTgChannels` 做一次兜底
- 或改为 `shallowRef` + 延迟计算，避免空数组导致的「无搜索源」误报

---

## 二、中优先级（中短期可做）

### 5. 搜索逻辑重复：任务创建模式

**现状**：`performParallelSearch`、`performFastSearch`、`performDeepSearch` 中任务构建逻辑高度相似（插件循环、TG 批次、AbortController、`$fetch` 等）。

**建议**：
- 抽取公共函数，例如 `createPluginTask(plugin, options)`、`createTgBatchTask(batch, options)`
- 统一请求参数和错误处理，减少重复和偏差

---

### 6. 控制台日志过多

**现状**：生产环境仍有 `[performParallelSearch] 开始执行`、`任务完成`、`当前聚合总数` 等 console 输出。

**建议**：
- 使用 `import.meta.dev` 或环境变量包装 `console.log`
- 或接入 `server/core/utils/logger.ts`，按 LOG_LEVEL 控制

---

### 7. 类型定义与共享

**现状**：`SearchResponse`、`MergedLinks` 等定义在 `server/core/types/models.ts`，composable 通过 `~/server/core/types/models` 引用。Nuxt 会自动处理，但语义上类型是服务端核心类型。

**建议**：
- 保持现状即可；若希望前后端更解耦，可考虑 `shared/types/` 或 `~/types` 作为共享类型入口

---

### 8. 单元测试覆盖 useSearch

**现状**：`test/unit/` 覆盖了 memoryCache、fetch、pluginManager、tg 等，但没有 `useSearch` 或客户端搜索流程的测试。

**建议**：
- 为 `extractMergedFromResponse` 编写纯函数测试
- 为 `mergeMergedByType` 编写合并逻辑测试
- 使用 `@vue/test-utils` 对 `useSearch` 做轻量集成测试（可 mock $fetch）

---

## 三、低优先级（长期或按需）

### 9. SearchService 职责拆分

**现状**：`SearchService` 同时负责 TG 搜索、插件搜索、缓存、合并、排序、熔断等，类体积较大。

**建议**（与 `docs/未来优化点.md` 一致）：
- 拆分为：`TgSearchService`、`PluginSearchService`、`ResultMergerService`、`SearchCoordinator`
- 分阶段逐步引入，避免大改影响稳定性

---

### 10. 插件注册简化

**现状**：存在 `globalRegistry` 和 `PluginManager` 双重抽象。

**建议**：参考 `docs/未来优化点.md` 中的说明，统一为 `PluginManager` 单一路径，移除全局注册表。

---

### 11. 错误与用户反馈

**现状**：任务失败时多为静默或 `console.warn`，用户感知有限；全部失败时仅有空状态提示。

**建议**：
- 增加「部分源失败」的轻度提示（如 toast）
- 在空状态下区分「无结果」与「所有源均失败」，给出不同文案
- 可配合服务端 `warnings` 字段做更精细的反馈

---

### 12. 流式/SSE 搜索（可选）

**现状**：通过多次独立请求 + 增量合并实现渐进展示，并非真正的 HTTP 流式。

**建议**（若需要真正的流式）：
- 新增 `/api/search-stream`，使用 `text/event-stream`
- 客户端用 `EventSource` 或 `fetch` + `ReadableStream`  incremental 解析
- 需要调整 SearchService 支持流式或增量回调

---

## 四、架构与数据流概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        客户端 (useSearch)                         │
├─────────────────────────────────────────────────────────────────┤
│  performSearch → performParallelSearch                           │
│       │                                                          │
│       ├─ 为每个插件 / 每批 TG 创建 task (含 AbortController)        │
│       ├─ p-limit 控制并发                                         │
│       ├─ 每批完成 → mergeMergedByType → setMerged                 │
│       └─ 支持 pause/continue (continue 走 performDeepSearch)       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ $fetch /api/search
┌─────────────────────────────────────────────────────────────────┐
│                        服务端 (SearchService)                     │
├─────────────────────────────────────────────────────────────────┤
│  search() → searchTG() || searchPlugins()                        │
│       │                                                          │
│       ├─ 缓存 (UnifiedCache)                                     │
│       ├─ 熔断 (PluginHealthChecker)                              │
│       └─ mergeResultsByType() → merged_by_type                   │
└─────────────────────────────────────────────────────────────────┘
```

---

## 五、建议执行顺序

1. 删除或接入 `performFastSearch`，统一搜索入口
2. 抽离 `extractMergedFromResponse` 并补充单测
3. 收敛 `continueSearch` 与 `performParallelSearch` 的语义
4. 屏蔽生产环境中的 console.log
5. 抽取任务创建公共函数，减少重复
6. 视需求推进 SearchService 拆分与 SSE 流式搜索
