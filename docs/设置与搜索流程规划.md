# 设置与搜索流程规划

## 一、问题诊断

### 1. 设置不生效：根因

**架构现状**：`app.vue` 和 `pages/index/index.vue` 各自调用 `useSettings()`，得到**相互独立**的 `settings` ref。

```
app.vue (useSettings)  →  settings A  ←  SettingsDrawer v-model
index.vue (useSettings) →  settings B  ←  getSearchOptions() 用于搜索
```

- 用户在设置抽屉中修改并保存 → 更新 `settings A`，并写入 localStorage
- 搜索时 `getSearchOptions()` 使用的是 `settings B`，仍为初次加载时的值
- 结论：搜索未使用最新设置，导致“只选一个 TG 频道却搜索了全部”

### 2. 是否还要传频道/插件参数

**结论**：仍需要传。

- 前端为每个源发起独立请求（插件 / TG 批次），每个请求必须明确指定要查的源
- 服务端是无状态 API，不知道“当前用户配置”，只能根据请求参数决定查哪些插件/频道
- 若改成“服务端读用户配置”，需要登录、会话或服务端持久化，改动较大

因此保留：**每个请求显式传入 `plugins=xxx` 或 `channels=xxx`**。

---

## 二、规划方案

### 1. 设置单例（高优先级）

**目标**：全应用共享同一份 `settings`，设置抽屉和搜索页使用同一数据源。

**做法**：在 `useSettings` 中使用模块级 ref，使所有调用者共享同一 ref：

```ts
// composables/useSettings.ts
const settings = ref<UserSettings>({...});  // 模块级，单例
export function useSettings() {
  return { settings, loadSettings, ... };   // 每次返回同一个 ref
}
```

**效果**：设置抽屉的修改会立即反映到搜索时的 `getSearchOptions()`，无需额外同步逻辑。

### 2. 搜索前强制同步（兜底）

在 `onSearch` 中，搜索前调用一次 `loadSettings()`，确保使用 localStorage 中的最新配置，避免因缓存或时序导致的旧数据。

### 3. 请求参数与请求模型（维持现状）

- 每个请求继续携带：`kw`、`src`（plugin/tg）、`plugins` 或 `channels`、`conc`、`ext` 等
- 前端按“选中的插件 + 选中的 TG 频道”构建任务，每个任务对应一次带明确参数的 API 请求
- 服务端按参数执行搜索，不维护用户配置

### 4. 尊重「只选 TG / 只选插件」的选择

**原逻辑**：`loadSettings` 中只要 `enabledPlugins` 为空，就强制恢复默认插件。

**问题**：用户只想用 TG 并清空插件时，会被强制加回插件，导致「只选一个 TG 却搜出全部」。

**修改**：仅在**插件和 TG 都为空**时恢复默认插件；否则保留用户选择（允许只选 TG 或只选插件）。

---

## 三、数据流（修复后）

```
用户修改设置 → SettingsDrawer emit → app.vue settings 更新
                    ↓
            saveSettings() 写入 localStorage
                    ↓
        settings 为单例，index 的 getSearchOptions 读取同一 ref
                    ↓
        performSearch(getSearchOptions()) 使用最新 enabledPlugins / enabledTgChannels
                    ↓
        为每个选中的插件/TG 批次创建任务，请求时传入对应 plugins/channels
```
